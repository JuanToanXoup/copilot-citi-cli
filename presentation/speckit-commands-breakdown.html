Spec-Kit Commands: Complete Breakdown

Spec-Kit provides six core commands that orchestrate the Specification-Driven Development workflow. Each command automates a critical phase of the development process, from establishing governance principles to generating executable code.

These commands transform a 12-hour manual specification process into a 15-minute automated workflow.

1
/speckit.constitution
Establish Architectural Governance
Purpose

Creates or updates the project's constitutional principles that govern all subsequent development decisions. This is the foundational step that ensures consistency, quality, and architectural integrity across all generated code.

ALWAYS FIRST: The constitution must be established before any other Spec-Kit commands. It defines the rules that all subsequent specifications, plans, and implementations must follow.
When to Use
  • At the very beginning of a new project (before any other commands)
  • When establishing organizational development standards
  • When refining architectural principles based on learnings
  • Before any feature specifications are created
Example Usage
/speckit.constitution Create principles focused on code quality, testing standards, user experience consistency, and performance requirements. Include governance for how these principles should guide technical decisions and implementation choices.
What It Does
1. Analyzes Requirements Interprets your governance needs and organizational constraints
2. Generates Constitution Creates structured articles covering architecture, testing, simplicity, etc.
3. Defines Gates Establishes checkpoints that all implementations must pass
4. Creates Amendment Process Documents how principles can evolve over time
Output
File Created: .specify/memory/constitution.md

Contains nine articles defining immutable principles:

  • Article I: Library-First Principle
  • Article II: CLI Interface Mandate
  • Article III: Test-First Imperative
  • Article VII: Simplicity and Minimal Complexity
  • Article VIII: Anti-Abstraction Principle
  • Article IX: Integration-First Testing
Key Constitutional Principles
Article Principle Enforcement
I Every feature must begin as a standalone library Modularity by default
II All functionality exposed via CLI (text in/out) Observability and testability
III No code before tests (strict TDD) Quality built-in from start
VII Maximum 3 projects initially Simplicity over complexity
VIII Use frameworks directly, no wrappers Avoid over-abstraction
IX Real databases over mocks Integration-first testing
Business Impact
Ensures: Consistent architecture across all teams, systematic quality enforcement, reduced technical debt, and maintainable codebases that follow proven patterns.
2
/speckit.specify
Create Feature Specification
Purpose

Transforms a natural language feature description into a complete, structured specification with automatic repository management. This is where business requirements become formalized, testable specifications.

When to Use
  • When starting a new feature or project
  • After constitutional principles are established
  • When you have a clear idea of WHAT to build (not HOW)
  • Before any technical planning or architecture decisions
Example Usage
/speckit.specify Real-time chat system with message history and user presence tracking. Users should be able to send messages, see who's online, and view conversation history. Support for 10K concurrent users.
Important: Focus on WHAT and WHY, not HOW. Do not mention tech stack, frameworks, or implementation details at this stage.
What It Does
1. Auto-Numbers Feature Scans existing specs to determine next feature number (001, 002, 003...)
2. Creates Branch Generates semantic branch name from description (e.g., "003-chat-system")
3. Generates Specification Copies and customizes feature template with your requirements
4. Structures Content Creates user stories, acceptance criteria, and non-functional requirements
Output
Files Created:
specs/003-chat-system/ └── spec.md # Complete feature specification

Git Branch Created: 003-chat-system

Business Impact
Reduces: Specification time from 2-3 hours to 5 minutes. Creates traceable, version-controlled requirements that stay in sync with code.
3
/speckit.clarify
Structured Specification Refinement
Purpose

Conducts sequential, coverage-based questioning to eliminate all ambiguities from the specification before technical planning begins. This critical step prevents downstream rework and ensures specifications are complete and unambiguous.

When to Use
  • Immediately after creating a specification with /speckit.specify
  • Before generating any implementation plans
  • When [NEEDS CLARIFICATION] markers exist in the spec
  • To validate completeness of requirements
Critical: This step is mandatory before planning. Skipping it leads to implementation plans with hidden assumptions and increased rework.
Example Usage
/speckit.clarify
Business Impact
Prevents: 80% of rework caused by ambiguous requirements. Ensures alignment between stakeholders before expensive implementation begins.
4
/speckit.plan
Generate Implementation Plan
Purpose

Transforms business requirements into a detailed technical implementation plan with architecture decisions, technology choices, and supporting documentation. This is where you specify the HOW after clarifying the WHAT.

When to Use
  • After specification is complete and clarified
  • When ready to make technical architecture decisions
  • After constitutional principles are established
  • Before task breakdown and implementation
Example Usage
/speckit.plan We are going to generate this using .NET Aspire, using Postgres as the database. The frontend should use Blazor server with drag-and-drop task boards, real-time updates. There should be a REST API created with a projects API, tasks API, and a notifications API.
Output
Files Created:
specs/003-chat-system/ ├── plan.md # High-level implementation strategy ├── research.md # Tech stack justifications, version details ├── data-model.md # Entity schemas, relationships ├── quickstart.md # Key validation scenarios └── contracts/ ├── api-spec.json # OpenAPI/REST contracts └── websocket-spec.md # Real-time communication contracts
Business Impact
Delivers: Complete technical architecture in 5 minutes vs. 3-4 hours manually. Every decision is documented, traceable, and constitutionally compliant.

/speckit.tasks
Generate Task Breakdown
Purpose

Analyzes the implementation plan and design documents to generate a detailed, dependency-ordered task list with parallelization markers. This creates the executable roadmap for implementation.

When to Use
  • After implementation plan is validated and approved
  • Before running /speckit.analyze (validation gate)
  • When you need a clear execution roadmap
  • To identify parallelizable work streams
Example Usage
/speckit.tasks
Output
File Created: specs/[feature]/tasks.md

Contains organized task breakdown by user story with dependency ordering and parallelization markers.

Business Impact
Provides: Clear execution roadmap with optimal task ordering. Enables parallel work streams and incremental delivery of user stories.
5.5
/speckit.analyze VALIDATION GATE
Cross-Artifact Consistency & Coverage Analysis
Purpose

Performs comprehensive validation of all specification artifacts to ensure consistency, completeness, and constitutional compliance. This is the final quality gate before code generation.

Critical Checkpoint: This validation step catches inconsistencies, gaps, and constitutional violations before expensive implementation begins. It's the last chance to fix specification issues without code rework.
When to Use
  • After /speckit.tasks has generated the task breakdown
  • Before /speckit.implement begins code generation
  • When you want to validate cross-artifact consistency
  • To ensure constitutional compliance across all documents
  • To verify test coverage will meet targets
Recommended: Always run this validation before implementation, especially for complex features or when multiple team members contributed to specifications.
Example Usage
/speckit.analyze

The command automatically runs in validation mode when spec.md, plan.md, and tasks.md exist.

What It Validates
1. Cross-Artifact Consistency Ensures spec.md, plan.md, contracts/, data-model.md, and tasks.md are aligned
2. Constitutional Compliance Validates all artifacts against the nine constitutional articles
3. Coverage Analysis Verifies tasks will achieve target test coverage (70-80%)
4. Completeness Check Identifies missing scenarios, edge cases, or acceptance criteria
5. Dependency Validation Ensures task ordering respects dependencies
6. Contract Alignment Verifies API contracts match user stories and data models
Validation Checks
Check Category What It Validates Common Issues Found
Specification Alignment User stories in spec.md match implementation phases in plan.md Missing user stories, orphaned implementation phases
Contract Consistency API contracts match acceptance criteria and data models Mismatched field names, missing endpoints, type conflicts
Data Model Integrity Entities in data-model.md support all user stories Missing relationships, undefined fields, circular dependencies
Task Coverage Tasks in tasks.md cover all acceptance criteria Untested scenarios, missing edge cases, incomplete flows
Test Strategy Test tasks will achieve 70-80% coverage target Insufficient test scenarios, missing integration tests
Constitutional Gates Plan passes Simplicity, Anti-Abstraction, Integration-First gates Over-engineering, unnecessary abstractions, mock-heavy tests

Validation Categories
Blockers

Critical issues that will cause implementation failure. Must be fixed before /speckit.implement.

  • Missing required functionality
  • Contract-spec mismatches
  • Insufficient test coverage
  • Invalid dependencies
Warnings

Issues that should be addressed but won't block implementation. May cause confusion or technical debt.

  • Naming inconsistencies
  • Constitutional violations
  • Incomplete error handling
  • Missing edge cases
Recommendations

Suggestions for improvement that enhance quality but aren't required.

  • Additional test scenarios
  • Performance optimizations
  • Enhanced documentation
  • Refactoring opportunities
Iterative Refinement Workflow
1. Run /speckit.analyze (validation mode) 2. Review validation-report.md 3. Fix blockers: - Update contracts to match specifications - Add missing test scenarios to tasks.md - Resolve data model inconsistencies 4. Address warnings: - Fix constitutional violations - Standardize naming conventions - Add error handling scenarios 5. Re-run /speckit.analyze to verify fixes 6. Proceed to /speckit.implement when validation passes
Best Practice: Treat validation as an iterative process. Each fix may reveal new issues. Continue until validation report shows zero blockers and minimal warnings.
Business Impact
Prevents: Implementation rework caused by specification inconsistencies, missing requirements, or constitutional violations. Catches issues in minutes that would take hours to fix in code.
ROI: 10-15 minutes of validation saves 2-4 hours of implementation rework. Typical validation finds 3-7 blockers and 5-12 warnings per feature.

/speckit.implement
Execute Implementation
Purpose

Executes the implementation plan by parsing the task breakdown and generating code in the correct order, respecting dependencies and following TDD practices. This is where specifications become working software.

When to Use
  • After all prerequisites are validated (constitution, spec, plan, tasks)
  • After /speckit.analyze (validation mode) passes with zero blockers
  • When ready to generate actual code
  • When local development environment is set up
Prerequisite: Always run /speckit.analyze (validation mode) before implementation to catch specification issues early.
Example Usage
/speckit.implement
Business Impact
Delivers: Working, tested code from specifications in minutes to hours vs. days to weeks. Follows TDD, constitutional principles, and best practices automatically.
